# 승원이의 빵빵 놀이터 — 발견/기획 초안

작성 시각: 2026-04-16 15:52 KST
상태: 요구사항 정리 중 / 구현 전

## 1. 프로젝트 목표

27개월 남자아기가 혼자서도 사용할 수 있는 초간단 학습 앱을 만든다.
핵심 목표는 글자와 숫자에 친숙해지게 하는 것이고, 조작은 거의 전부 탭만 허용한다.
앱은 Play Store 배포가 아니라 APK 개인 설치용으로 만든다.

## 2. 현재까지 확정된 핵심 요구사항

### 플랫폼
- Android APK 개인 설치용
- Galaxy S24 우선
- 다른 안드로이드 폰/태블릿에서도 동작하게 설계
- 가로모드 고정
- 인터넷/로그인/광고/개인정보 기능 없음
- 시스템 차원은 완전 키오스크가 아니라 immersive full-screen 중심의 2단계 잠금 전략

### 사용자 경험
- 아이 혼자 주로 사용, 가끔 보호자 지도
- 한 화면에 큰 버튼 1~2개 수준으로 단순화
- 뒤로가기/종료/복잡한 메뉴는 숨김
- 슬라이드/드래그 없이 탭만 사용
- 정답 후 자동으로 다음 문제로 이동
- 시간 제한/패널티/목숨 시스템 없음

### 앱 구조
- 첫 화면: 카테고리 선택(한글 / 알파벳 / 숫자)
- 각 카테고리 내부에서 학습 모드와 게임 모드 분리
- 히어로 페이지 존재
- 자동차가 달리는 느낌의 진입 연출 선호
- 진입 연출 뒤 로고와 플레이 버튼 노출
- 앱 이름: 승원이의 빵빵 놀이터

### 비주얼/사운드
- 파스텔 톤
- 자동차/타요 감성
- 심플하지만 아기자기한 UI
- 음성은 반드시 필요
- 여성 목소리 선호
- 한글 세트/알파벳 세트 내 음성 톤 일관 유지
- 효과음은 풍성한 게임 스타일
- 배경음악은 학습/게임 중 계속 재생
- 실제 동요 기반 BGM 10곡 구성 희망

### 캐릭터/자산 방향
- 실제 아기 사진을 히어로 연출에 합성하는 구상 있음
- 타요 캐릭터/이미지/분위기 사용 희망
- 참고 앱: 별별한글
- 현재는 private-use 개인용 전제를 두고 설계하되, 자산은 나중에 교체 가능한 구조로 분리하는 것이 바람직함

## 3. 콘텐츠별 요구사항

### 3-1. 한글
목표 우선순위가 가장 높음.

확정 내용:
- 1차는 자음/모음 중심
- 쌍자음/복모음은 후반 레벨
- 음절은 더 높은 레벨에서 도입
- 자음 음성은 "기역, ㄱ"처럼 이름+글자 함께 제시
- 모음은 입모양/혀 위치 느낌으로 시각 설명 강화
- 학습 순서:
  1) 자음
  2) 모음
  3) 쌍자음
  4) 복모음
  5) 음절
- 게임은 처음부터 4지선다
- 큰 카드 + 탭 재생 + 소리 듣고 맞히기 중심
- 그림 설명 중요

권장 레벨 구조(초안):
- Level 1: 기본 자음 14
- Level 2: 기본 모음 10
- Level 3: 쌍자음 5
- Level 4: 복모음 11
- Level 5+: 쉬운 음절 묶음(가/나/다 등 또는 자모 결합형)

### 3-2. 알파벳
확정 내용:
- 대문자/소문자 함께 제시
- 퀴즈는 대소문자 섞어서 출제
- 음성은 letter name 기준
- 대문자/소문자 구분 없이 같은 소리로 읽음 (A/a 모두 에이)
- 1단계: 소리 듣고 글자 맞히기
- 이후 확장:
  - 대문자 보고 소문자 찾기
  - 소문자 보고 대문자 찾기
  - 짝맞추기
- 단어 연결은 다음 단계로 미룸

### 3-3. 숫자/수학
확정 내용:
- 숫자 학습 카드 자체는 생략 가능
- 바로 게임형 수학 개념으로 진입
- 개수 세기, 순서 맞추기, 크다/작다, 간단한 부등호, 덧셈/뺄셈을 단계적으로 제공
- 숫자 자체 음성보다 문제 안내 음성이 중요
- 개수 비교는 자동차 개수와 숫자를 함께 놓고 비교 → 개수만 비교 → 숫자만 비교 순으로 확장 가능
- 개수 비교는 20개 이하까지
- 숫자 비교는 단계별로 3자리 수까지 확장 가능
- 덧셈/뺄셈 1차는 1~10 범위

## 4. 게임 UX 요구사항

- 기본 문제 형식은 4지선다
- 드래그 없음, 탭만
- 틀렸을 때:
  - 화면 흔들림
  - X 표시
- 맞혔을 때:
  - 축하 연출
  - 자동차 스티커 보상
- 세트 완료 시:
  - 폭죽/반짝이
  - 자동차 스티커 획득
- 매 문제마다 큰 축하보다 세트 단위 연출 선호

실무 기본값 제안:
- 세트당 5문제
- 세트 종료 후 스티커 1개 지급
- 연속 정답 보너스는 초기 버전에 생략

## 5. 보호자 기능 요구사항

확정 내용:
- 보호자 메뉴 진입: 특정 아이콘 5번 탭
- 보고 싶은 정보: 카테고리별 헷갈리는 것
- 틀린 것만 다시 5문제 기능 필요
- 진도 바꾸기 필요

추가로 기본 포함 추천:
- 앱 종료 버튼(보호자 메뉴 안에만)
- 레벨 잠금/해제
- 자산/사운드 볼륨 간단 토글(후순위)

## 6. 구현 기본 방향 제안

현재 요구사항 기준으로는 Flutter가 가장 무난하다.
이유:
- Android APK 빌드가 간단함
- 휴대폰/태블릿 대응이 쉬움
- 가로모드 고정, 전체화면, 로컬 에셋 관리가 편함
- 데이터 중심 콘텐츠 확장(JSON/asset 파일) 설계가 쉬움

추천 아키텍처 초안:
- Flutter 앱
- 로컬 JSON 기반 콘텐츠 정의
- 로컬 asset 폴더에 음성/이미지/BGM 저장
- 로컬 저장소(shared_preferences 또는 Hive/Isar)로 진도/오답 기록 저장
- 카테고리/레벨/문항/보상 모두 데이터 기반으로 분리

## 7. 아직 남아 있는 핵심 결정 포인트

이 문서 작성 시점 기준 미확정 또는 기본값 제안 상태:

1. 한글 레벨별 정확한 학습 목록과 순서 최종 확정
2. 모음 입모양 설명 자산을 어떤 스타일로 만들지(벡터/실사진/참고 앱 준용)
3. 세트당 문제 수 최종값(기본 제안 5)
4. 레벨 해금 규칙
   - 보호자 수동 전환만 허용할지
   - 자동 해금도 허용할지
5. 히어로 연출 길이 및 스킵 가능 여부
6. 실제 아기 사진의 사용 범위
   - 히어로 연출만
   - 스티커/프로필에도 활용
7. 타요 이미지/BGM/사진 자산 전달 방식
8. 숫자 연산 문제의 시각 형식
   - 자동차 그림 + 식 같이 표시
   - 그림만 먼저, 식은 후반부
9. 보호자 메뉴에서 허용할 설정 항목 범위
10. 참고 앱 별별한글에서 구체적으로 참고할 요소

## 8. 구현 착수 전 추천 순서

1. 요구사항 잔여 항목 잠금
2. 기술 스택 최종 결정(기본안: Flutter)
3. 콘텐츠 데이터 스키마 설계
4. 정보구조/화면 흐름 설계
5. 첫 프로토타입 범위 확정
6. 앱 기본 골격 생성
7. 한글/알파벳/숫자 각 1세트씩 vertical slice 구현
8. APK 빌드 및 기기 테스트

## 9. 첫 구현 범위 제안 (현재 기준)

첫 playable 버전은 아래를 목표로 잡는 것이 적절하다.

- 히어로/홈 화면
- 한글:
  - 기본 자음 카드 학습
  - 소리 듣고 맞히는 4지선다
- 알파벳:
  - A/a 세트부터 시작하는 대소문자 통합 카드/퀴즈
- 숫자:
  - 1~5 또는 1~10 범위의 개수 비교/간단한 덧셈 뺄셈
- 보호자 메뉴:
  - 진도 변경
  - 오답 5문제 재도전
  - 카테고리별 헷갈리는 항목 보기

## 10. Git 운영 원칙

사용자 요청에 따라 이 저장소에 작은 단위로 커밋/푸시하면서 진행한다.
권장 방식:
- docs/설계 먼저 커밋
- 기본 앱 스캐폴드 커밋
- 카테고리별 기능을 작은 단위로 커밋
